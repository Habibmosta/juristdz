Bonne question stratégique. Allons au fond.

C'est quoi exactement ces modèles
AraBERT — BERT pré-entraîné sur du texte arabe (Wikipedia arabe, actualités). Produit des embeddings de 768 dimensions qui "comprennent" la sémantique arabe. Variante intéressante : AraBERT-legal fine-tuné sur des textes juridiques arabes.

CamemBERT-legal — BERT français fine-tuné sur des décisions de justice françaises (Cour de cassation, Conseil d'État). Comprend le vocabulaire juridique français : "attendu que", "infirme le jugement", "moyen de cassation", etc.

Gemini text-embedding-004 — Le modèle d'embedding de Google. Multilingue, 768 dimensions, supporte l'arabe et le français nativement, accessible via API.

Compatibilité entre eux
Oui et non — et c'est le point crucial.

Les embeddings de AraBERT et ceux de Gemini ne sont pas dans le même espace vectoriel. Tu ne peux pas mélanger un embedding AraBERT avec un embedding Gemini et faire une recherche cosine cohérente. C'est comme comparer des coordonnées GPS avec des coordonnées cartésiennes — même nombre de dimensions, résultats absurdes.

Il faut choisir un seul modèle d'embedding et l'appliquer à tout le corpus.

Les 3 architectures possibles pour nous
Option A — Gemini Embeddings uniquement (recommandée pour démarrer)

Texte FR/AR → Gemini text-embedding-004 → vecteur 768d → pgvector
Avantages :

Déjà dans notre stack (on utilise Gemini)
Multilingue natif, arabe + français dans le même espace vectoriel
Pas de serveur à gérer, API simple
Coût : ~$0.00002 / 1000 tokens (quasi gratuit)
Inconvénients :

Moins spécialisé juridiquement que CamemBERT-legal
Dépendance Google
Option B — CamemBERT-legal pour le français + AraBERT pour l'arabe

Texte FR → CamemBERT-legal → vecteur 768d → index pgvector FR
Texte AR → AraBERT → vecteur 768d → index pgvector AR
Requête → détection langue → embedding correspondant → recherche dans l'index correspondant
Avantages :

Meilleure qualité sémantique juridique
Comprend vraiment "cassation", "pourvoi", "moyen" en FR et leurs équivalents AR
Inconvénients :

Deux index séparés, requête bilingue complexe
Il faut héberger les modèles (Python/FastAPI sur un VPS ou HuggingFace Inference API)
Les modèles font 400-500MB chacun
Latence plus élevée
Option C — Architecture hybride (la meilleure à terme)

Indexation : CamemBERT-legal (FR) + AraBERT (AR) → deux index pgvector
Requête utilisateur → Gemini pour reformulation/expansion → embeddings spécialisés → fusion des résultats
Gemini joue le rôle de "query understanding" — il reformule, expand les synonymes, détecte l'intention. Les modèles spécialisés font la recherche vectorielle. C'est ce que font les systèmes avancés.

// services/embeddingService.ts
const GEMINI_EMBEDDING_URL = 
  'https://generativelanguage.googleapis.com/v1beta/models/text-embedding-004:embedContent';

export async function getEmbedding(text: string): Promise<number[]> {
  const res = await fetch(`${GEMINI_EMBEDDING_URL}?key=${API_KEY}`, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      model: 'models/text-embedding-004',
      content: { parts: [{ text }] },
      taskType: 'RETRIEVAL_DOCUMENT'
    })
  });
  const data = await res.json();
  return data.embedding.values; // 768 floats
}

Étape 4 — Fonction RPC de recherche sémantique

CREATE OR REPLACE FUNCTION search_jurisprudence_semantic(
  query_embedding vector(768),
  similarity_threshold float DEFAULT 0.7,
  match_count int DEFAULT 20
)
RETURNS TABLE (
  id uuid, case_number text, summary_fr text,
  legal_domain text, decision_date date, similarity float
)
LANGUAGE sql STABLE AS $$
  SELECT id, case_number, summary_fr, legal_domain, decision_date,
    1 - (embedding <=> query_embedding) AS similarity
  FROM jurisprudence
  WHERE status = 'validated'
    AND 1 - (embedding <=> query_embedding) > similarity_threshold
  ORDER BY embedding <=> query_embedding
  LIMIT match_count;
$$;
